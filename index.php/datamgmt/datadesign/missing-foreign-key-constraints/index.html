Missing foreign key constraints

by George Mastros (gmmastros) on November 20, 2009 in category Data Modelling and Design. Article views: 158,480
Tweet Instapaper

References are at the heart of a database. It is possible to create a beautiful database with perfectly working front end code that always, 100% of the time, does the right thing with your data. But, writing code is hard. Very hard! Your data is often the most important asset you own. You need to protect it with every bit of technology you can find. At the heart of protecting your data is referential integrity. What does this mean? It means that you shouldn’t be missing data, ever!

The code below will check for columns that have ID in the name of the column where that column is not part of a primary key or foreign key constraint. Often times, this represents a missing constraint, but not always. The code presented below exists to highlight potential problems. You must still determine if this potential problem is real, and then act accordingly.

How to detect this problem:

SELECT  C.TABLE_SCHEMA,C.TABLE_NAME,C.COLUMN_NAME
FROM    INFORMATION_SCHEMA.COLUMNS C          
        INNER Join INFORMATION_SCHEMA.TABLES T            
          ON C.TABLE_NAME = T.TABLE_NAME    
          And T.TABLE_TYPE = 'Base Table'
          AND T.TABLE_SCHEMA = C.TABLE_SCHEMA        
        LEFT Join INFORMATION_SCHEMA.CONSTRAINT_COLUMN_USAGE U            
          ON C.TABLE_NAME = U.TABLE_NAME            
          And C.COLUMN_NAME = U.COLUMN_NAME
          And U.TABLE_SCHEMA = C.TABLE_SCHEMA
WHERE   U.COLUMN_NAME IS Null          
        And C.COLUMN_NAME Like '%id'
ORDER BY C.TABLE_SCHEMA, C.TABLE_NAME, C.COLUMN_NAME

How to correct it: Correcting this problem seems simple at first. Just declare your foreign keys, right? Well, it’s not so simple. You see, there could be code running that deletes all the necessary data from the related tables. If you have code that deletes data in related tables in the wrong order, you will get referential constraint errors. Similar problems can occur with updates and inserts. The order in which you do things is important when you have referential constraints.

Level of severity: High

Level of difficulty: High

    Here is the version for PostgreSQL (where pg_catalog schema is used):

    —-
    SELECT n.nspname AS schema,
    c.relname AS table_name,
    a.attname AS column_name,
    t.conname AS key_name
    FROM pg_catalog.pg_attribute a
    INNER JOIN pg_catalog.pg_class c
    ON (a.attrelid = c.oid AND c.relkind = ‘r’)
    INNER JOIN pg_catalog.pg_namespace n
    ON (n.oid = c.relnamespace)
    LEFT JOIN pg_catalog.pg_constraint t
    ON (a.attrelid = t.conrelid
    AND t.conkey[1] = a.attnum
    AND t.confrelid != 0)
    WHERE a.attnum > 0
    AND a.attname LIKE ‘%Id’
    AND t.conname IS NULL
    ORDER BY schema, table_name, column_name;
    ——

    What about unique constraints/index!? Columns that them self has a unique constraint/index should not be warned about. I found a nice script at Mark on Databases, etc. where you get a sample hint.
    The code below will check for column names that ends with id or code, where that column is not part of a primary key, unique constraint or foreign key constraint.

    Find missing foreign key constraints

    <code lang="lang-sql">
select a.*
from (
	select c.table_schema as "Schema", c.table_name as "Table", c.column_name as "Column"
		from information_schema.columns c
		inner join information_schema.tables t on c.table_name = t.table_name
			and t.table_type = 'base table'
			and t.table_schema = c.table_schema
		left join information_schema.constraint_column_usage u on c.table_name = u.table_name
			and c.column_name = u.column_name
			and u.table_schema = c.table_schema
		where u.column_name is null and lower(c.column_name) like '%id' or lower(c.column_name) like '%code'
) as a
full join (
	select "Schema" = object_schema_name(si.object_id),	st.name as "Table",	sc.name as "Column"
		from sys.indexes si 
		inner join sys.index_columns ic on si.object_id = ic.object_id
			and si.index_id = ic.index_id
			and si.is_unique = 1
			and si.is_primary_key = 0
		inner join sys.columns sc on ic.object_id = sc.object_id
			and ic.column_id = sc.column_id 
		inner join sys.tables st on si.object_id = st.object_id 
) as b on a."Schema" = b."Schema"
	and a."Table" = b."Table"
	and a."Column" = b."Column"
where b."Schema" is null
order by a."Schema", a."Table", a."Column";
</code>
