When do statistics update?

by Ted Krueger (onpnt) on January 28, 2013 in category Database Administration Microsoft SQL Server Microsoft SQL Server Admin. Article views: 7,157
Tweet Instapaper

The question is often asked: when will SQL Server statistics update if auto update stats is enabled?

The short answer: When auto update stats is enabled in a database, statistics will update when 20% + 500 rows have changed in the table.Â  This change can be adding new rows, removing rows or updating rows.

If you attend sessions or read many tuning articles that involve statistics on the internet, you may have seen the statement, â20% + 500 rowsâ more than a few times.Â  Some related information on when the 20% + 500 does actually come into play and how the cardinality of the table plays a role can be found in KB 195565, âStatistical maintenance functionality (autostats) in SQL Serverâ.

Specifically a section extractedâ¦

The basic algorithm for auto update statistics is:

  • If the cardinality for a table is less than six and the table is in the tempdb database, auto update with every six modifications to the table.
  • If the cardinality for a table is greater than 6, but less than or equal to 500, update status every 500 modifications.
  • If the cardinality for a table is greater than 500, update statistics when (500 + 20 percent of the table) changes have occurred.

For table variables, cardinality changes does not trigger auto update statistics.

Referenced from – http://support.microsoft.com/kb/195565

The best way to look at this is to give it a try and see if the statement is accurate.

Taking a closer look

Statistics are the lifeline of generating an efficient method for retrieving data by the optimizer.Â  Statistics will base the cost in a form of the estimated amount of data that will be retrieved.Â  This could mean the difference between operations such as physical join operations leading to sorts or inadequate estimation of memory allocation needs.Â  If statistics are outdated or missing, execution plans can be inaccurate and cause severe performance issues.

Since statistics are so critical to cost estimation and plan generation, knowing how and when statistics are updated is just as critical.Â  Of course, this knowledge isnât just wasted space in the mix of knowing how SQL Server works but gives a person in charge of maintaining a database power to know how to maintain statistics correctly.

Given statistics will be automatically updated when 20% + 500 rows have changed, we can estimate based on the total number of rows or growth expectancy, when statistics may be outdated and cause a potential issue.Â  Imagine a table that has 5000 rows of data in it, and that data changes often.Â  This would indicate 5,000 * .2 + 500 = 1,500 rows would have to change before statistics would be updated.Â  1,500 rows to 5,000 isnât truly a great deal of data when the 5,000 is being changed at a high rate.Â  Now, think of a table that has 1,000,000 rows in it.Â  This would equate to 1,000,000 * .2 + 500 = 200,500 rows before statistics will update.Â  200,500 rows is a much larger number and if it took a long time to reach that, but possibly reaches 190,000 quickly, we potentially have an issue and statistics could be poorly representing the data in the table.

Summary

Statistics are a crucial factor in how SQL Server and the Optimizer comes to an Execution Plan it will use to retrieve data.Â  Knowing how configurations such as autostat and others that directly affect how statistics are updated, stored and what they are composed of, is also a crucial factor so we can maintain them, make better decision on them and troubleshoot them when potential performance issues arise.
